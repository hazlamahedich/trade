## What Went Well

### 1. Organic Knowledge Compound Effect

team mantis a identified this as the standout pattern: knowledge is growing organically across epics. Each story's Dev Notes explicitly references 2-3 previous stories' intelligence. Story 3-4 referenced learnings from 3-1, 3-2, and 3-3. This cumulative knowledge transfer eliminated guesswork and rework.

**Impact:** Zero integration failures, 3x scope expansion handled smoothly.

**Evidence:** Every story file in Epic 3 has a "Previous Story Intelligence" section.

### 2. Party-Mode Adversarial Pre-Review Scaled to 4 Stories

Stories 3-3, 3-4, 3-5-3, and 3-6 underwent multi-agent adversarial review BEFORE implementation. The most impactful outcome was Story 3-3's **unanimous 4-agent consensus to strip Redis caching** — the GROUP BY benchmark proved it unnecessary.

**Impact:** Saved ~30 test cases, ~6 hours implementation time, and avoided config patching complexity across all existing tests.

**Evidence:** Story 3-3 Dev Notes document the full adversarial review rationale with 4 numbered arguments.

### 3. Prep Sprint Foundation Was Rock-Solid

Every Epic 2 retro action item was completed (9/10, deployment pending). The prep sprint delivered Vote model, DebateRepository, rate limiter, and Alembic migration — all critical Epic 3 dependencies. Zero foundation work was needed during the epic itself.

**Impact:** Epic 3 started at Story 3-1 with all infrastructure in place.

**Evidence:** Story 3-1's "What Already Exists" table lists 7 pre-built components.

### 4. Unplanned Scope Handled Gracefully

Three stories (3-5-1, 3-5-2, 3-5-3) were born from implementation reviews, not upfront planning. Each was scoped, reviewed, and delivered without disrupting the planned story sequence.

**Impact:** Product quality improved mid-flight. Rounding overflow, WS/poll race condition, and optimistic update UX gap all addressed proactively.

**Evidence:** Stories 3-5-1 through 3-5-3 all trace to "Story 3-4 party-mode review" in their Change Logs.

### 5. First Voter Celebration — Adversarial Review Caught 7 Bugs Pre-Code

Story 3-6's adversarial review by Winston, Sally, Amelia, and Murat identified 7 issues (2 CRITICAL, 2 HIGH) before implementation, including `useRef` not triggering re-renders (BUG-1) and missing timeout cleanup (BUG-2).

**Impact:** Implementation was clean — no debug issues logged.

**Evidence:** Story 3-6 "Adversarial Review Findings" table documents all 12 pre-implementation fixes.

## What Didn't Go Well

### 1. DebateStream God Component (450+ Lines)

`DebateStream.tsx` grew from ~376 lines to 450+ across Epic 3. Every story added "just a few lines" — vote controls wiring, WS vote handler, optimistic state, first voter hook. Acknowledged in Story 3-2 party review but deferred.

**Impact:** Maintenance burden. Every future story that touches the debate page will add more wiring.

**Status:** Action item for pre-Epic 4 prep.

### 2. Accessibility Findings Are Predictable and Recurring

Same 4 categories appeared in ALL frontend story reviews:
- Reduced motion (`prefers-reduced-motion`)
- `aria-live` regions for screen readers
- Touch target sizing
- Contrast ratios

~80% of a11y review findings could have been caught by a pre-submission checklist.

**Impact:** ~15+ review cycles spent on predictable issues across 4 stories.

**Status:** Frontend a11y checklist to be created as action item.

### 3. No Production Deployment (3 Epics Overdue)

Deployment scripts are ready but Railway/Vercel accounts never set up. This has been on every retro since Epic 1.

**Impact:** Cannot validate NFR-01 (latency) under real conditions. Cannot validate Guardian accuracy with real traffic. Three epics of code exist only locally.

**Status:** CRITICAL action item. Charlie owns it.

### 4. Test Quality 'B-to-A' Pattern

Initial TEA scores consistently land in 78–88 range, then improve to 91–95 after review. The review process works, but initial submission quality should be higher given we're 3 epics in.

**Impact:** Review cycles spent on preventable quality gaps.

**Evidence:** Story 3-4 started at 78/100, improved to 92/100 after fixes.

### 5. Capacity Model Measures Wrong Thing

`VOTE_CAPACITY_LIMIT` measures votes-per-minute, not unique active voters. Backlog item B1 (switch to Redis SET/SADD) is architecturally important but keeps getting deferred.

**Impact:** NFR-04 compliance is based on throughput, not concurrent users.

**Status:** MEDIUM priority action item.

## Significant Discoveries

1. **Organic Knowledge Compound Effect** — team mantis a's key insight. Each epic builds exponentially on previous epics' documented learnings. Story Dev Notes explicitly reference 2-3 previous stories' intelligence. This is the primary velocity accelerator and should be protected as a process.

2. **Redis Caching Rejected via Adversarial Consensus** — Story 3-3's party-mode review achieved unanimous 4-agent consensus to strip an entire architectural layer because benchmarking proved the optimized GROUP BY was sufficient (~3-5ms). Most impactful scope-reduction decision in project history.

3. **Component Size Debt Is Real** — `DebateStream` grew from ~376 to 450+ lines. Every story added "just a few lines" of wiring. Individually justified, collectively problematic. The 300-line component limit is the corrective action.

4. **Accessibility Findings Are Predictable** — Same 4 categories in every frontend review: reduced motion, aria-live, touch targets, contrast. Pre-submission checklist would eliminate ~80% of review cycles on these items.

5. **Test Quality 'B-to-A' Pattern** — Initial TEA scores consistently land 78–88, improve to 91–95 after review. Review process works but initial quality should target 85+ floor.
