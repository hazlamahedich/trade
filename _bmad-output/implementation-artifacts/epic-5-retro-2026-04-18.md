## Key Insights

1. **Spike-before-implement should be a formal rule** for any epic touching unfamiliar libraries or API boundaries. One upfront investigation prevented failures across all 4 stories.

2. **Large stories need phase checkpoints, not splitting.** The detail in 85-subtask stories contributed to high execution quality. The problem was the *experience* of working through them — explicit "stop and validate" checkpoints within the story would help.

3. **A canonical `renderWithProviders()` test utility** would eliminate the cascade tax. Consolidate scattered test render helpers into one shared utility. Adding a new provider becomes a one-line change, not a 7-file fix.

4. **Detail-oriented story creation is a strength.** The subtask count is a symptom of thoroughness, not bloat. Don't reduce detail — improve the workflow for consuming it.

5. **Team follow-through culture is strong.** 70%→100% retro follow-through rate across two epics. The team trusts that action items will be executed.

## Lessons Learned (Candidates for AGENTS.md)

### 25. Spike-First for Unfamiliar Libraries

**Pattern:** Story 5-1 spike validated Satori constraints (variable fonts crash, static TTF only, inline styles only) before any production code. This knowledge prevented failures across all 4 stories.

**Rule:** Any epic that touches an unfamiliar library, API, or external service MUST start with a pre-dev spike. Document constraints, validate assumptions, then implement. The spike file becomes reference material for all subsequent stories.

### 26. Phase Checkpoints Within Large Stories

**Pattern:** Stories 5-1 (41), 5-2 (56), 5-3 (~85) all exceeded the 30-subtask threshold. Flagging alone didn't help the implementation experience. The detail contributed to high quality (scores 82-96) but working through 85 subtasks without a midpoint was exhausting.

**Rule:** Stories exceeding 30 subtasks must include explicit phase checkpoints (e.g., "validate after subtask 40"). This preserves the detail while giving implementers natural stop-validate-continue moments. Do NOT reduce detail — improve the workflow for consuming it.

### 27. Canonical Test Render Helper

**Pattern:** Every frontend story that adds Radix UI or Framer Motion components breaks existing test files. Story 5-3 broke 7 DebateStream test files. Each story pays a "cascade tax" of fixing N broken tests before writing new ones.

**Rule:** ALL test files MUST use a shared `renderWithProviders()` utility from the test setup. This helper wraps all commonly-needed providers (TooltipProvider, MotionProvider, QueryClient, etc.). Adding a new provider is a one-line change to the shared utility, not an N-file hunt. NEVER use raw `render()` directly.
