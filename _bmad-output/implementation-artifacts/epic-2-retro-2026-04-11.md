## What Went Well

### 1. Adversarial Pre-Review (Party-Mode) Was a Game-Changer

Stories 2-3 and 2-4 underwent party-mode adversarial spec review with 4 agents reviewing specs BEFORE implementation. This caught architectural issues, edge cases, and dependency gaps before any code was written.

**Impact:** Zero integration failures between stories. Cross-story dependency chain (2-1 → 2-2 → 2-3 → 2-4 → 2-5) held perfectly.

**Evidence:** Stories 2-3 and 2-4 had the cleanest implementations despite being the most complex.

### 2. Discriminated Union State Modeling

The `useGuardianFreeze` hook used TypeScript discriminated unions (`idle | active | acknowledged`) instead of boolean flags. This eliminated impossible states at the type level and made the freeze/unfreeze lifecycle impossible to get wrong.

**Evidence:** Zero state-related bugs in Story 2-3 testing (score: 92).

### 3. Defense-in-Depth Sanitization

Two-layer approach: (1) prompt-level prohibition telling the LLM not to use forbidden phrases, and (2) regex safety-net catching anything that slips through. The prompt layer catches ~90% of cases; the regex catches the rest.

**Evidence:** `SanitizationResult` and `SanitizationContext` Pydantic models provide structured audit trail. 100% of forbidden phrases caught in testing.

### 4. Reused `asyncio.Event` Pattern from Story 1-6

The coordination pattern from Stale Data Guard was adapted for Guardian interrupt flow with minimal changes. This validated Epic 1's forward-thinking architecture.

**Evidence:** Story 2-2 Dev Notes explicitly reference Story 1-6's `asyncio.Event` pattern.

## What Didn't Go Well

### 1. Token Streaming Leak (Story 2-4)

Forbidden phrases are briefly visible during streaming before sanitization completes. This is a known limitation documented in Story 2-4.

**Impact:** Users may see restricted text for a fraction of a second during token streaming.

**Status:** Acknowledged as known limitation. Fix requires architecture change (pre-buffer tokens before emitting).

### 2. No Production Deployment (Carried from Epic 1)

Epic 1's deployment scripts are ready but Railway/Vercel accounts were never set up. Epic 2 code exists only locally.

**Impact:** Guardian moderation accuracy can't be validated with real traffic. NFR-01 (latency) not measured under real conditions.

### 3. Missing E2E Integration Coverage

While unit tests are strong (82–98 scores), there's no end-to-end Playwright test covering the full Guardian flow (debate start → interrupt → freeze → resume → completion).

**Impact:** Cross-cutting issues between backend Guardian logic and frontend freeze UI could go undetected.

## Significant Discoveries

1. **State rebuild bug in `engine.py`** — The main debate loop silently dropped `guardian_interrupts` and `pause_history` when rebuilding `current_state` each turn. This was a real production bug that would have caused Guardian interrupt data loss in long debates. Found by Task 1 E2E validation.

2. **Token streaming leak was architecture-level** — The original per-chunk sanitization approach was fundamentally flawed (forbidden phrases straddle chunk boundaries). Required full rewrite to buffer-all-sanitize-whole-emit-delta pattern.

3. **Pydantic config patching requires all fields** — `app.config.settings` is a Pydantic model that validates on attribute access. Partial mocks cause validation errors. Documented in AGENTS.md for all future tests.
